# 毕业设计答辩讲解稿

## 基于社区数据反馈的电商广告推荐系统的设计与实现

---

### 开场（约30秒）

各位老师好，我是XX，我的毕业设计题目是《基于社区数据反馈的电商广告推荐系统的设计与实现》。下面我将从研究背景、系统设计、核心实现和测试结果四个方面进行汇报。

---

### 一、研究背景与问题（约2分钟）

电商平台的广告推荐面临一个核心矛盾：平台想多投广告来赚钱，但广告太多用户就会流失。传统的广告频控机制对所有用户"一刀切"——统一限制每人每天看多少条广告，这种做法忽略了一个事实：活跃用户对广告的容忍度远高于低活跃用户。

我的研究就是要解决这个问题。具体思路是：在电商平台中引入社区子系统，通过用户在社区中的评价、问答、点赞等行为来量化用户的活跃度，然后根据活跃度等级实施差异化的广告频控策略——活跃用户多看广告提升收入，低活跃用户少看广告保护留存。

这样做的创新之处在于：把社区行为数据作为广告策略的决策信号，形成了"社区参与→活跃度提升→广告策略优化→留存提升"的正向闭环。

---

### 二、系统总体设计（约3分钟）

系统采用前后端分离的单体分层架构，包含五个核心模块：

**第一，电商业务模块。** 提供商品管理、订单系统、用户系统等基础电商功能，同时实时记录用户的浏览、搜索、加购、购买等行为数据。

**第二，社区子系统。** 支持商品评价（1-5星评分+文字评价）和商品问答两种交互方式。社区行为是活跃度计算的重要数据来源。

**第三，推荐引擎。** 采用业界标准的多阶段漏斗架构：召回层使用UserCF、ItemCF、Content-Based、ALS矩阵分解和热门召回五种算法多路并行；排序层使用DeepFM和DIN两个深度学习模型精细打分；重排层通过MMR算法保证推荐多样性。

**第四，广告系统。** 实现了eCPM竞价排序、CPC/CPM两种计费模式和预算控制机制。广告的展示决策需要通过频控组件的审核。

**第五，活跃度引擎。** 这是连接社区数据和广告频控的桥梁。

关于数据流：用户行为产生后写入行为日志表，活跃度引擎读取近30天的行为，按权重乘以时间衰减因子累加得到活跃度评分，根据评分划分等级，频控组件据此确定广告展示参数。

---

### 三、核心创新——频控机制（约3分钟）

这是我这个系统最核心的部分，我重点讲解。

**活跃度评分公式：**

score = Σ（行为权重 × e^(-0.1 × 距今天数)）

行为权重的设计：购买+10分，评价和回答+5分，加购+3分，登录+2分，点赞+2分，浏览和搜索+1分。可以看到社区行为的权重很高——发表一条评价等于浏览5个商品，这就是激励用户参与社区的机制。

时间衰减函数用的是指数衰减：今天的行为权重为1，7天前约0.5，30天前只有0.05。这保证评分反映的是用户近期的活跃状态。

**三级频控策略：**

- 高活跃用户（≥60分）：每页3条广告，最小间隔60秒，每日上限50条
- 普通用户（20-60分）：每页2条广告，最小间隔120秒，每日上限30条
- 低活跃用户（<20分）：每页1条广告，最小间隔300秒，每日上限10条

**频控判断流程：**

每次要展示广告时，系统先查用户活跃度等级确定参数，再检查今日已展示次数是否达到上限，再检查距上次展示是否满足最小间隔，三个条件都满足才展示广告。

这样设计的效果是：一个经常发评价、频繁购物的用户，活跃度会很高，系统判定他对平台很忠诚，可以多给他看广告来提升收入；而一个很少登录的用户，活跃度很低，系统判定他随时可能流失，就减少广告打扰来保住他。

### 四、技术实现要点（约2分钟）

**后端：** Python FastAPI框架，SQLAlchemy ORM，JWT认证，SQLite数据库。共实现了40多个REST API接口。

**推荐算法：** 协同过滤用scikit-learn的cosine_similarity实现；DeepFM和DIN用PyTorch自研实现，DeepFM结合FM层和DNN层捕获低阶和高阶特征交叉，DIN用注意力机制动态建模用户兴趣。

**前端：** Vue 3 + Element Plus + ECharts，UI设计融合了天猫、京东、淘宝三大平台的风格。管理后台提供活跃度分布饼图、广告效果排行图、频控策略效果图等可视化分析。

**广告混排展示：** 首页每3个推荐商品穿插1条广告，以原生广告形式展现。商品详情页也有广告推荐区。广告展示和点击事件实时上报，用于计费和效果统计。

### 五、测试结果（约1分钟）

系统共通过77个测试用例的验证：

- 单元测试48个：覆盖所有推荐算法、活跃度评分、频控逻辑、竞价计费
- API测试21个：覆盖所有接口的正常和异常场景
- 集成测试6个：验证完整的购买流程、社区互动流程、广告频控流程等端到端场景
- 性能测试2个：推荐接口响应时间<500ms，商品列表接口<200ms

种子数据生成了1000件中文商品、100个用户、12000条行为记录，可以完整演示系统功能。

### 六、总结（约1分钟）

总结一下本系统的三个创新点：

1. **社区驱动的频控机制。** 把社区行为数据引入广告频控决策，这是区别于传统方法的核心创新。

2. **差异化广告策略。** 根据用户活跃度三级分层，实施差异化的广告参数，而不是统一阈值。

3. **正向激励循环。** 社区行为在活跃度计算中的高权重设计，形成了社区参与→活跃度提升→体验优化→留存提升的正向循环。

不足之处：推荐模型缺乏在线学习能力，频控参数基于经验设定而非自动优化，未来可引入强化学习自动搜索最优参数。

以上就是我的汇报，请各位老师批评指正。

### 预设答辩问题及回答

**Q1：为什么选择指数衰减函数而不是其他衰减方式？**

指数衰减有三个优点：一是函数值域为(0,1]，今天的行为权重恰好为1，数学性质好；二是衰减速度由单一参数λ控制，调参简单；三是函数连续平滑，不会出现阶梯式跳变。相比线性衰减，指数衰减更符合"近期行为重要性远大于远期"的直觉——7天前的行为权重就衰减了一半，30天前基本归零。

**Q2：频控策略的参数（60分/20分的阈值，每页广告数等）是怎么确定的？**

阈值的设定基于行为权重的分析：一个每天登录并浏览10个商品的用户，30天的活跃度约为 (2+10)×Σdecay ≈ 12×6.6 ≈ 79分，属于高活跃；一个每周登录一次只浏览几个商品的用户约为 (2+3)×4周×衰减 ≈ 15分，属于低活跃。60和20的划分使三个等级的用户分布比较均衡。具体的广告条数和间隔参考了行业通用做法，未来可以通过A/B测试进一步优化。

**Q3：DeepFM和DIN的区别是什么，为什么都要用？**

DeepFM擅长捕获特征之间的交叉关系，包括用户属性、商品属性和上下文特征的低阶和高阶组合。DIN则专门针对用户行为序列建模——它通过注意力机制，根据不同的候选商品动态激活用户历史中的相关兴趣。两者互补：DeepFM看的是静态特征组合，DIN看的是动态兴趣演化。实际系统中可以将两者的输出做加权融合。

**Q4：如果一个用户突然变得不活跃了，频控策略多久能反应过来？**

由于时间衰减函数的特性，如果用户停止活跃，其评分会自然衰减：7天不活跃评分衰减约50%，14天衰减约75%。一个原本60分的高活跃用户，如果完全停止使用，大约10天左右会降到普通级别，20天左右降到低活跃级别。系统每次请求时实时计算活跃度，所以反应是即时的，不存在延迟。

**Q5：这个系统和实际电商平台的推荐系统有什么差距？**

主要差距有三点：一是数据规模，实际平台面对亿级用户和千万级商品，需要分布式架构和近线计算，我的系统是单机演示级别；二是实时性，实际平台用Kafka+Flink做实时特征计算和模型更新，我的系统是请求时实时计算；三是策略优化，实际平台有完善的A/B测试和自动调参机制，我的频控参数是经验值。但在架构设计思路和算法原理上是一致的。

**Q6：社区行为作为活跃度信号，会不会被恶意刷量利用？**

这是一个好问题。确实存在用户为了提升活跃度而灌水的风险。应对措施可以包括：对同一用户短时间内的重复行为做去重或降权；对评价内容做质量检测（过短、无意义的评价不计分）；设置单日单类行为的贡献上限。本系统作为演示项目未实现这些反作弊机制，但在实际部署中是必须考虑的。
